SailAway Project
Nonfunctional Requirements

Number Requirement Type Definition Source
# A1 Application Architecture No matter how the SailAway application is developed, the design must support the separation of the user interface, the event model, and the physical model. The Library must be treated as a separate application, so it's design is not within the scope of this project. Interworld Transport Management

# P1 Physical Model Interface The physical model must be able to accept commands to iterate forward or backward along an orbit one step at a time. Commands for multiple steps or stepping while or until conditions are met are properly part of the user interface. Interworld Transport Management
# P2 Physical Model Interface The physical model must publish a basic set of commands to which it will respond when so ordered by the event model. These commands do not need to be in a spoken language since it is the responsibility of the user interface to correctly link spoken language commands to the physical model through the event model. Interworld Transport Management
# P3 Physical Model Interface Some elements within a sail model will serve special purposes that may include elements of the following list. These elements should be nameable so the user interface may display them by name in graphical displays.
sail bus sail fabric payload batteries transmitters
receivers antennae docking adaptors tension lines computing electronics
pulleys servos cameras general sensors thermal controls
Interworld Transport Management

# U1 User Interface Any GUI used by the Student or Supporter must be able to present a very simplified picture of the Sail Model with the ability to switch on and off a variety of graphical aids showing derived data such as sun angle, clock angle, position, speed, time elapsed and other secondary information. Interworld Transport Management
# U2 User Interface Any UI used by the Researcher or Developer must be able to present all internally used primary properties of all objects in a graphical or XML format to the user.

The UI must also be able to present all secondary, derived data for these same users in an XML format as an information dump.

Interworld Transport Management
# U3 User Interface The user commands must be writable in a spoken language. The system must be able to parse the commands to the internal command set represented as a collection of Events. This requirement is mainly to support languages other than English. Interworld Transport Management
# U4 User Interface It must be possible to plug in user interfaces into the system as replacements for the one that comes packaged with SailAway. These alternate UI's will function by adhering to the API of the physical model

In order to meet this requirement, it may be necessary to place each of the application packages into different jar files. It is probably true that the physical model will have to be separated out at a minimum.

Interworld Transport Management
# U5 User Interface Every user interface capable of storing or retrieving storable objects must be able to handle a flat XML formatted file. The minimal demands for library interaction involve passing XML files that represent the contents ofthe storable objects, so each UI that can save or retrieve must be able to handle these at a minimum. Interworld Transport Management
# U6 User Interface The developer's UI should be have a very slim profile. A menu, tool bar, and status bar should be enough. The user would show the windows to objects they intend to view from menu or toolbar choices they make. These separate windows would stay in touch with the main interface and update as various commands were completed. Interworld Transport Management
# U7 User Interface The researcher's UI should be a collection of windows into the various objects they have up and active as working objects. These windows would demonstrate to which objects they belong and what remains to be done with them. The constructive hints from validators would appear at the bottom of these viewers. An enclosed desktop approach to containing these viewers is the desired one. Interworld Transport Management
# U8 User Interface The Supporter's UI should keep to a rule of not allowing creation of new objects through the UI. Instead, it should focus upon giving a good presentation for objects already in the library. It should also make it very easy to run short 'templated' command sequences on those existing objects. These templates would be determined by what outside observers tend to focus their questoins upon. A well developed FAQ will be needed before this requirement is fleshed out. Interworld Transport Management
# U9 User Interface The Notification Tool is assumed to be a separate application like email, IRC or some other messaging system. No real requirements are being made for covering all types of messaging system, but email will be expected as the default option. The system must be able to format outgoing messages according to the protocol type used by the messaging system. Because of this, the System must understand how to construct a basic email message or hand the information to an object that can do the work and comes as a standard on all machines that can run SailAway. Interworld Transport Management

# E1 Event Model The process that performs the saving of anything to a library will register with all storable objects to receive event information needed to notice these changes. Interworld Transport Management

# L1 Library Interface SailAway will be packaged with the addresses of Library sites willing to provide remote library services. The default configuration of SailAway will be to use remote libraries listed with the package. Interworld Transport Management
# L2 Library Interface The system must be able to package scheduled flight states for delivery to remote libraries. A package would be delivered for storage in one transaction and treated by the library as an aggregate to be authorized in one swoop. Interworld Transport Management
# L3 Library Interface When an application delivers or receives a storable object from a remote library, the object will be transferable as an XML file. All GUI's that interact with libraries will be able to accept the flat file format, so each library must be able to handle XML representations of objects at a minimum.

It is perfectly acceptable if the library stores the objects in a different format, however. A librarian may choose to store serialized objects, tabular data, or any other format they wish. They may also be willing to transfer storable objects in formats other than XML. As long as the library can handle the one basic format that is assumed useable by all the advanced UI's, any extra features offered by the Librarian are their own business.

Interworld Transport Management
# L4 Library Interface All transactions between the System and a Library must be secured in the sense that that both sides must be able to identify the source and target user of the request. It is not necessary that the transaction be secured in the cryptographic sense yet because there are no passwords involved. This kind of security is mostly a naming scheme, but will eventually evolve into a higher security system to support the creation of research groups with interests in closing their research long enough to publish in a competitive environment. Interworld Transport Management
