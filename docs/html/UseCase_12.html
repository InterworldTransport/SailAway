Name Validate Simulation to be run
Iteration Finished
Summary When the System is ordered to run a simulation, it will determine whether the offered simulation is well formed or not. Any simulation that passes this test is handed on to another process within the system for further work. Any simulation that does not pass this test is analyzed for why it failed and useful error messages are passed back to the part of the system that called here.
Basic Course of Events
  • New or changed information about a simulated flight is entered into the system.
  • The System finds or forms a test queue for determining whether the simulation is well-formed or not.
  • The System hands the simulation and test queue to a process that runs the tests. (See Use Case 'Run a validation test'.)
  • If the test results are negative, a copy of the test result is handed to the Notification Tool for delivery to the user.
  • The System exits validation mode by returning the test results to the calling process.
Alternative Paths If the System decides the simulation is not well formed, the test that failed will deliver some explanatory text to the calling process explaining what failed and why.
Exception Paths .
Extension Points .
Trigger The System triggers this case after a user has altered a simulation in any way that might change wether it is well-formed.

This case is also triggered when a simulated flight is firt run if no information is available concerning whether the simulation is well-formed.

Assumptions .
Preconditions There must be a simulation whether it is well formed or not for this validator to do anything.
Postconditions This validator will stay out of the way and not cause any further interactions within the System after its initial job of validating a new or changed flight simulation is done. When it is done, the System will act upon the validity information provided by this validator.
